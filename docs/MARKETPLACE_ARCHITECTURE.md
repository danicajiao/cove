# Marketplace Architecture: SQL & REST

> **Note:** This is a planned backend architecture, not the current implementation. The app currently uses Firebase (Firestore + Auth + Storage) exclusively. This document describes the target design as Cove migrates to a custom backend — see [Backend Infrastructure](BACKEND_INFRASTRUCTURE.md) for the migration phases.

## Overview

Cove is a local marketplace discovery app where users browse vendors and products and track places they want to visit. Purchases happen in person — there is no order management or payment processing in v1.

The target backend uses:
- **PostgreSQL** for all structured data.
- **REST APIs** (OpenAPI-specified Go services) consumed by the iOS app via a typed Swift client generated by `swift-openapi-generator`.
- **Firebase Auth** retained throughout — ID tokens are validated at the gateway on every request.

> **Out of scope for v1:** Orders, payments, fulfillment, reviews, comments, social feed, GraphQL, and Firebase Auth replacement.

---

## Services

| Service | Responsibility |
|---|---|
| `cove-gateway` | Single ingress point. Validates Firebase ID tokens, routes to backend services. |
| `cove-product` | Product catalog, categories, search (Postgres FTS). |
| `cove-user` | User profiles, favorites, visit list. |
| `cove-image` | Image upload and serving via imgproxy + MinIO. |

Each service has its own OpenAPI spec. The iOS app uses `swift-openapi-generator` to produce a typed Swift client per service.

---

## Data Model (v1)

All entities live in PostgreSQL.

| Entity | Service | Notes |
|---|---|---|
| User profiles | `cove-user` | Structured, relational. UID sourced from Firebase Auth. |
| Product listings | `cove-product` | Structured, supports relationships (vendor, category). |
| Vendor management | `cove-product` | Relational, strong consistency. |
| Favorites | `cove-user` | Per-user product saves; synced from Firestore subcollection in migration. |
| Visit List | `cove-user` | Per-user rows with status tracking (`pending` → `visited`). |

## Example: Fetching a Product Page

The iOS app calls `cove-product` through `cove-gateway`:

```
GET /products/{id}
Authorization: Bearer <Firebase ID Token>
```

```swift
// Generated by swift-openapi-generator from cove-product's OpenAPI spec
let response = try await productClient.getProduct(.init(path: .init(id: productId)))
let product = try response.ok.body.json
```

## Database Schema (PostgreSQL)

```sql
-- Users table
-- UID matches the Firebase Auth UID so no join is needed to validate identity.
CREATE TABLE users (
    uid VARCHAR(128) PRIMARY KEY,   -- Firebase Auth UID
    username VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Vendors table
CREATE TABLE vendors (
    id SERIAL PRIMARY KEY,
    owner_uid VARCHAR(128) NOT NULL,
    business_name VARCHAR(255) NOT NULL,
    description TEXT,
    FOREIGN KEY (owner_uid) REFERENCES users(uid)
);

-- Categories table
CREATE TABLE categories (
    id VARCHAR(128) PRIMARY KEY,    -- matches Firestore categoryId during migration
    name VARCHAR(100) NOT NULL
);

-- Products table
CREATE TABLE products (
    id VARCHAR(128) PRIMARY KEY,    -- matches Firestore productId during migration
    vendor_id INT NOT NULL,
    category_id VARCHAR(128) NOT NULL,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    price DECIMAL(10, 2),
    image_key TEXT,                 -- MinIO object key, served via cove-image
    search_vector TSVECTOR,         -- populated by trigger for Postgres FTS
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (vendor_id) REFERENCES vendors(id),
    FOREIGN KEY (category_id) REFERENCES categories(id)
);

CREATE INDEX products_search_idx ON products USING GIN(search_vector);

-- Favorites table
CREATE TABLE favorites (
    uid VARCHAR(128) NOT NULL,
    product_id VARCHAR(128) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (uid, product_id),
    FOREIGN KEY (uid) REFERENCES users(uid),
    FOREIGN KEY (product_id) REFERENCES products(id)
);

-- Visit list table
CREATE TABLE visit_list (
    id SERIAL PRIMARY KEY,
    uid VARCHAR(128) NOT NULL,
    type VARCHAR(10) NOT NULL CHECK (type IN ('vendor', 'product')),
    vendor_id INT,
    product_id VARCHAR(128),
    status VARCHAR(10) NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'visited')),
    purchased BOOLEAN NOT NULL DEFAULT FALSE,
    notes TEXT,
    added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    visited_at TIMESTAMP,
    FOREIGN KEY (uid) REFERENCES users(uid),
    FOREIGN KEY (vendor_id) REFERENCES vendors(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);
```

## iOS Networking Layer

The iOS app talks to the backend through a repository protocol layer introduced in the Phase 0 refactor. Each repository has a Firebase implementation (current) and a Remote implementation (target):

```
ProductRepository (protocol)
├── FirebaseProductRepository   ← current (Firestore)
└── RemoteProductRepository     ← target (cove-product REST API)

UserRepository (protocol)
├── FirebaseUserRepository      ← current (Firestore)
└── RemoteUserRepository        ← target (cove-user REST API)

FavoritesRepository (protocol)
├── FirebaseFavoritesRepository ← current (Firestore subcollection)
└── RemoteFavoritesRepository   ← target (cove-user REST API)

ImageRepository (protocol)
├── FirebaseImageRepository     ← current (Firebase Storage)
└── RemoteImageRepository       ← target (cove-image REST API)
```

ViewModels depend only on the protocol — swapping the implementation requires a one-line change at the injection site.

## Deployment

See [Backend Infrastructure](BACKEND_INFRASTRUCTURE.md) for the full K3s deployment model. Summary:

- Services run as pods in `cove-staging` and `cove-prod` namespaces on a home K3s cluster
- Argo CD reconciles manifests from the `homelab` repo
- Cloudflare Tunnel exposes the cluster without open ports
- Secrets via External Secrets Operator pulling from GCP Secret Manager
